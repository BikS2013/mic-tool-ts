# Issues — Pending Items

This file tracks open issues, pending follow-ups, and dependency-vetting decisions for the `mic-tool-ts` project. Pending items are listed first (most critical at the top), followed by completed items, followed by the dependency-vetting log.

## Pending Items

- **Implement the other seven LLM providers** (severity: feature gap; plan-003 v2) — `openai`, `anthropic`, `google`, `azure-ai-inference`, `ollama`, `litellm`, `openai-compat`. All seven are accepted by config validation but throw `LLMConfigurationError` at refiner construction via `src/llm/factory.ts`. Each needs its own `<provider>.ts` adapter under `src/llm/`, wiring in `factory.ts`, and provider-specific env-var resolution in `resolveProviderConfig()` (`src/config.ts`). The error messages already document the env vars each provider expects (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`, `AZURE_AI_INFERENCE_*`, `OLLAMA_HOST`, `LITELLM_*`, `OPENAI_COMPAT_*`).

- **Port microphone capture to Linux and Windows** (severity: feature gap; investigation-007) — `src/mic/index.ts` currently returns `SoxMicSource` only on `process.platform === "darwin"` and throws `UnsupportedPlatformError` for `linux` and `win32`; tests explicitly assert that non-macOS platforms fail. Investigation in `docs/reference/investigation-007-portability-windows-linux.md` found the STT, LLM, renderer, config, and orchestrator layers are mostly portable, but the project needs Linux/Windows mic backends behind the existing `MicSource` contract. Recommended first path: keep macOS on SoX, add FFmpeg `pulse`/`alsa` support for Linux and FFmpeg `dshow` support for Windows, then add per-OS mocked unit tests and manual smoke scripts under `test_scripts/`.

- **Add `AZURE_OPENAI_API_KEY_EXPIRES_AT` support in code** (severity: medium) — Per CLAUDE.md `<configuration-guide>` the Azure OpenAI key, being a credential that expires, should have an analogous renewal reminder. Currently only `SONIOX_API_KEY_EXPIRES_AT` is wired (`src/config/expiry.ts` → `warnAboutExpiry()` called from `src/main.ts`). The configuration guide documents the variable as "not yet read by code"; this issue tracks closing that gap. Suggested approach: generalize `evaluateExpiry()` / `warnAboutExpiry()` to take a `{ envName, isoDate, renewUrl }` triple and call it once per tracked credential.

- **Live end-to-end run with a real Soniox API key + microphone** (severity: blocker for release, not for review) — AC-5, AC-6, AC-9, AC-11 from `docs/design/refined-request-soniox-mic-transcriber.md` require a real Soniox session, real audio, and macOS mic permission. All static-code preconditions (units A–E, plus turn detection, plus refinement, plus four-tier env chain) are in place; only the live verification is missing.

- **Live end-to-end run with a real ElevenLabs API key + microphone** (severity: blocker for ElevenLabs release validation, not for review) — The ElevenLabs provider is covered by mocked WebSocket unit tests and static verification, but it still needs a real `ELEVENLABS_API_KEY`, accepted Scribe terms in the ElevenLabs dashboard, real microphone audio, and macOS mic permission to validate live partial/final behavior against the production API.

- **Optional shell test wrappers under `test_scripts/`** (severity: minor; AC-14) — `test_scripts/test-help.sh`, `test_scripts/test-version.sh`, `test_scripts/test-missing-key.sh`. The existing `test_scripts/` folder contains TypeScript sanity scripts (`sanity-config.ts`, `sanity-mic.ts`, `sanity-renderer.ts`) but the AC-14 shell wrappers are not yet authored. Each should invoke the built binary, capture stdout/stderr/exit code, and assert.

- **Renderer `dispose()` write-after-end risk** (severity: minor) — `StdoutRenderer.dispose()` writes `\x1b[2K\r` to `process.stdout` even after the orchestrator's teardown has decided to exit. In overwrite mode this is intentional (shell-prompt hygiene) and safe because `process.stdout` is a synchronous TTY write. Documented here so a future change to a non-TTY-aware write path doesn't accidentally write after `process.exit`.

## Completed Items

- **Voice protocol status command** (resolved 2026-05-16) — Issue: users needed a spoken `command status` command to report current voice-agent settings during a live session. Solution: added `status.reported` to the protocol event model, taught the state machine to treat `command status` as a non-mutating report instead of an unknown operator, report `refine` / `translate` / `clipboard` state, `translation_policy`, and `pending_section`, render a human-readable status line in dictation/hybrid modes, emit JSONL in agent-protocol/hybrid modes, and documented the behavior in README, tool docs, project functions, and project design. Added focused protocol tests for state-machine status, pending-section status, JSONL output, and dictation rendering.

- **Wrapped overwrite transcript rows repeated stale text** (resolved 2026-05-16) — Issue: in TTY `overwrite` mode, a partial transcript longer than the terminal width could wrap across multiple physical rows, but the renderer only used `\r` and single-line padding. Subsequent partial updates could not clear the wrapped rows above the cursor, leaving repeated or stale transcript text as the STT provider emitted new snapshots. Solution: `StdoutRenderer` now tracks the physical row count of the previous overwrite snapshot, clears all previously occupied rows with ANSI cursor movement only on the TTY overwrite path, repaints the latest snapshot once, and resets the row state on final output / refinement / dispose. Added focused renderer tests for wrapped partial-to-partial and partial-to-final repainting; `pnpm typecheck`, focused renderer tests, and the full test suite pass.

- **Repeated interim transcript snapshots rendered as duplicate text** (resolved 2026-05-16) — Issue: realtime STT providers can emit the same partial hypothesis multiple times before a final commit, and Soniox live result frames can also repeat the current finalized prefix while the non-final suffix is still evolving. The previous `SonioxTranscriber` logic appended every final token run unconditionally, so phrases like `Εσύ, ρε παιδί μου... δείξ` could be duplicated inside the same rendered partial many times. Solution: added Soniox finalized-prefix overlap merging, kept renderer-level suppression of identical consecutive partial snapshots, reset the suppression cache on final output / turn boundaries / refined output / dispose, added focused Soniox and renderer regression tests, and updated renderer/provider design and user docs.

- **Voice protocol submit marker changed from `command end` to `command send`** (resolved 2026-05-16) — Issue: the spoken section-submit marker needed to be `command send` instead of `command end`. Solution: changed the default `MIC_TOOL_TS_SECTION_END_PHRASE` value to `command send`; updated protocol/config/orchestrator tests and all protocol documentation references in README, tool docs, configuration guide, project design, project functions, and plan-007.

- **Voice protocol marker wording changed from slash markers to command-prefixed phrases** (resolved 2026-05-16) — Issue: the implemented defaults used slash-style markers (`/command`, `/end`, `/cancel`), but the desired spoken protocol is `command refine|translate|clipboard`, `command send`, and `command cancel`. Solution: changed default marker config to `command`, `command send`, and `command cancel`; updated the state machine so `command <operator>` enables the operator and `command <operator> off` disables it; updated protocol tests, config tests, README, configuration guide, tool docs, project design, project functions, and plan-007.

- **Voice section-processing protocol** (resolved 2026-05-16) — Issue: `plan-007` described an explicit oral protocol for agent communication, but the implementation still only had guard-phrase turn detection. Solution: added the `src/protocol/` layer with command-prefixed marker matching, section state machine, JSONL writer, and controller; added `--interaction-mode`, marker phrase flags, operator defaults, translation policy, and hybrid `--protocol-output`; wired finalized STT text through the protocol controller; added unit coverage for marker matching, state changes, submission, cancellation, JSONL sequencing, operator processing, clipboard events, and shutdown cancellation; updated README, configuration guide, tool docs, project design, and project functions.

- **ElevenLabs transcription provider** (resolved 2026-05-16) — Issue: the tool only supported Soniox for realtime transcription, while the user wanted to know whether ElevenLabs could be supported and used as an alternative. Solution: researched ElevenLabs Scribe Realtime v2, documented the refined request and investigation, added `--stt-provider soniox|elevenlabs`, added provider-specific `ELEVENLABS_API_KEY` and expiry config, introduced `src/transcription/` provider dispatch, implemented `src/elevenlabs/client.ts` over the realtime WebSocket API, updated docs, and added mocked WebSocket tests.
