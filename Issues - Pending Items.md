- **Renderer `dispose()` write-after-end risk** (severity: minor) — `StdoutRenderer.dispose()` writes `\x1b[2K\r` to `process.stdout` even after the orchestrator's teardown has decided to exit. In overwrite mode this is intentional (shell-prompt hygiene) and safe because `process.stdout` is a synchronous TTY write. Documented here so a future change to a non-TTY-aware write path doesn't accidentally write after `process.exit`.

- **Unit C (Soniox client wrapper — `src/soniox/client.ts`)** — implemented 2026-05-16. `pnpm typecheck` passes.
  - **SDK surface used** (cross-checked against `node_modules/@soniox/node/dist/index.d.mts`): client class is `SonioxNodeClient` (the research doc TL;DR mentions `SonioxClient` in one place — the actual SDK only exports `SonioxNodeClient`); constructor option key is `api_key` (snake_case); session factory is `client.realtime.stt(config, options)`; error classes `AuthError`, `BadRequestError`, `QuotaError`, `ConnectionError`, `NetworkError`, `AbortError`, `StateError`, `RealtimeError` are all named exports and all extend `RealtimeError` (except the base itself). The `'error'` event handler signature is `(error: Error) => void` per `SttSessionEvents`.
  - **Partial/final algorithm chosen** (single source of truth = `'result'` event; the SDK's per-token `'token'` event is intentionally ignored to avoid duplicate processing):
    1. Drop tokens whose `text === "<end>"` or `text === "<fin>"`.
    2. For each `'result'`: append non-final token texts to a running `partialBuffer`; append final token texts to a per-result `finalText` accumulator.
    3. After processing the result, if `partialBuffer.length > 0` emit `onPartial(partialBuffer)`.
    4. If the result contained any final tokens, emit `onFinal((finalText + partialBuffer).trim())` and reset `partialBuffer = ""`.
    5. The `'finalized'` event flushes any residual `partialBuffer` as a final (defence-in-depth; normally a no-op because `finalize()` causes the server to emit final tokens that step 4 already handled).
    6. The `'endpoint'` event is only logged under `--verbose`; the partial→final promotion in step 4 already commits the utterance.
  - **Shutdown sequence**: `finalize()` → 250 ms drain → `Promise.race(finish(), reject-after-1500 ms)` → on reject/timeout, `close()`. Stays under AC-8's 1.5 s budget.
  - **Connect timeout**: overridden to **5000 ms** (SDK default 20000) so AC-10 network-failure paths fail fast.
  - **Unsolicited disconnect** (i.e. `'disconnected'` fired without `stop()` being called) is mapped to `SonioxNetworkError("Soniox connection lost: <reason>")` and dispatched via `onError`.
  - **Synchronous `sendAudio()` errors** (e.g. `StateError` from a race between the `state === "connected"` guard and the send) are caught, mapped via `mapSdkError`, and dispatched via `onError` — never allowed to crash the process.
  - **Deviations from the research doc**:
    - Research §6 mixes `'token'` and `'result'` listener patterns; we picked `'result'` only and documented the choice in a code comment. Matches the canonical algorithm in `project-design.md §6.5`.
    - The research doc's alternative "track a `pendingText`" simpler approach was NOT adopted; the chosen algorithm follows the project-design.md spec exactly.
    - Marker filter is implemented at the wrapper boundary only; Unit D will add a defence-in-depth second filter per project-design.md §7.3.
  - **Interface note for Unit E orchestrator**: this wrapper exposes the stub's `onPartial/onFinal/onError` setters, NOT the unified `start(callbacks)` shape described in `project-design.md §3.3`. The orchestrator must register the callbacks via the setters before calling `start()`. The `onEndpoint` and `onClose` callbacks from the design doc are not in this stub contract; the wrapper folds endpoint-driven commits into the standard final-promotion path and maps unsolicited disconnects to `onError(SonioxNetworkError(...))`.
- **Unit E (Orchestrator — `src/main.ts`)** — implemented 2026-05-16. `pnpm typecheck` clean. `main(argv): Promise<number>` resolves config, builds the `StdoutRenderer`, constructs `SonioxTranscriber`, wires `onPartial → renderer.partial`, `onFinal → renderer.final` and `onError → recordAsyncError + shutdown("transcriber-error")`, then opens the Soniox session via `transcriber.start()` BEFORE spawning the mic. On `transcriber.start()` failure we dispose the renderer and return `handleTopLevelError(err)`. On `createMicSource()` or `mic.start()` failure (after Soniox is connected) we call `await transcriber.stop()` so the WS is closed cleanly before returning the error. Once the mic is running we wire `mic.audio.on("data") → transcriber.pushAudio`, `mic.audio.on("error") → shutdown("mic-error")`, and `mic.audio.on("end") → shutdown("mic-end")`. SIGINT/SIGTERM listeners trigger the same idempotent `shutdown(reason)` (sequence: `mic.stop()` → `transcriber.stop()` → `renderer.dispose()`); a second SIGINT during shutdown writes `[mic-tool] force quit` and calls `process.exit(130)` so the user can always escape a hung close (SIGTERM mirrors with exit 143). `main()` parks on a `shutdownPromise` resolved by the teardown chain, then strips the signal handlers in a `finally` block before returning. `handleTopLevelError(err)` prints `<code>: <message>` to stderr for `MicToolError` (returning `err.exitCode`), the full `err.stack` for unexpected `Error` (return 1), or `String(err)` for anything else (return 1). `HelpOrVersionShown` is caught and mapped to exit 0 without printing. Verbose mode emits a start-of-session summary (`outputMode`, `language`, platform, node version, then "Listening on default mic. Press Ctrl+C to stop.") and tagged teardown diagnostics — never the API key value. The orchestrator never calls `process.exit` itself except in the force-quit escape; `src/index.ts` performs the final `process.exit(code)`. Smoke checks performed: `pnpm dev --help` → exit 0 with the full usage block; `pnpm dev --version` → exit 0 printing `0.1.0`; `pnpm dev` with no key in any source → exit 2 with `missing_configuration: SONIOX_API_KEY is not set...` on stderr.
